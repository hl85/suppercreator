## 一、内聚性审计

### 问题 1 — `url-to-markdown` 与 `danger-x-to-markdown` 功能重叠（高）

`url-to-markdown` 内置了 X/Twitter 适配器，`danger-x-to-markdown` 同样处理 X URL，并额外支持媒体下载。用户提供 `x.com` 链接时路由不明确，两个 skill 的差异（官方适配 vs 逆向 + 媒体下载）被埋在命名前缀中而非文档说明里。

**建议**：在两个 skill 的 description 中明确写出各自适用场景，或在 `docs/visuals.md` 中添加决策表。

---

### 问题 2 — 四个"文章 → 图片" skill 缺决策指引（中）

| Skill | 输出 | 典型场景 |
|-------|------|----------|
| `cover-image` | 1 张封面图 | 文章需要头图 |
| `article-illustrator` | N 张定位内联图 | 文章需要章节配图 |
| `infographic` | 1 张信息密集摘要图 | 复杂数据需可视化 |
| `xhs-images` | 2–10 张可滑动卡片 | 分发目标是小红书 |

这四个 skill 在任何文档中都未被并排比较，用户需要逐一阅读 SKILL.md 才能选择。

**建议**：在 `docs/visuals.md` 中增加上述决策表。

---

### 问题 3 — `post-to-wechat` 混合了三种工作流（中）

单一 skill 捆绑了：API 发文章、浏览器发文章、浏览器发图文。三种路径的认证方式、输入格式、错误模式互不相同，违反单一职责原则。

**建议**：在 SKILL.md 开头用三段式导航说明各路径，并各给一个最简示例。

---

### 问题 4 — `danger-` 前缀是孤例约定（中）

全库仅两个 skill 使用 `danger-` 前缀，`docs/creating-skills.md` 未定义此约定。该前缀在路由时像警告信号，导致用户跳过合法 skill。`post-to-x` 同样使用 Chrome CDP 却没有此前缀，进一步造成不一致。

**建议**：将"非官方 API"说明移入 description body，统一命名风格（见"行动建议 P2"）。

---

### 问题 5 — v0.1.0 与 v1.56+ 版本割裂无视觉信号（中）

6 个新 skill（`content-review`、`idea-radar`、`multi-publish`、`post-analytics`、`writeflow`、`markdown-to-thread`）处于实验阶段但无任何标识，与稳定 skill 在目录中并列显示，用户无从区分。

---

### 问题 6 — `xhs-images` 使用平台缩写命名（低）

同系列的 `post-to-wechat`、`post-to-weibo` 都使用全平台名，`xhs-images` 是唯一例外。

**建议**：重命名为 `xhs-images`，保留 `xhs`、`小红书` 作为 description trigger。

## 二、易用性审计

### 问题 7 — v0.1.0 Skills 展示了无法执行的 CLI 命令（高）

`content-review`、`idea-radar`、`writeflow` 等 6 个 skill 本质上是 prompt 驱动，但 SKILL.md 中写有 `./sc-run <skill> <command>` 示例。用户实际运行后会得到脚本未找到错误，严重损害信任。

**建议**：在这些 skill 的 description 和用法章节顶部添加：

```
> ⚠️ Beta — 此 skill 为 prompt 驱动，通过 Claude 对话调用，以下 CLI 命令尚未实现。
```

---

### 问题 8 — `multi-publish` 命令格式与全库不一致（高）

其他 23 个 skill 均使用 `./sc-run <skill> <script> [args]`，而 `multi-publish` SKILL.md 中展示的是：

```
/multi-publish <article.md> --to <wechat,x,weibo>
```

这是 Claude skill 调用格式，不是 CLI 格式，破坏了用户的操作习惯。

**建议**：统一改为 `./sc-run multi-publish main article.md --to wechat,x,weibo`。

---

### 问题 9 — `article-illustrator` → `imagine` 两步管道不直观（中）

工作流要求：

```bash
# Step 1
./sc-run article-illustrator build-batch --input article.md
# → 生成 output-dir/prompts/batch.json

# Step 2（需用户手动触发）
./sc-run imagine build-batch --batchfile output-dir/prompts/batch.json
```

两个 SKILL.md 均未解释中间文件 `batch.json` 的作用，也未说明为何需要两步。用户预期一次调用完成。

**建议**：在 `article-illustrator` SKILL.md 末尾添加"下一步"章节，说明如何将输出交给 `imagine`。

---

### 问题 10 — 无新用户入门路径（中）

24 个 skill 各自文档完善，但没有一处将完整飞轮组装成可跟随的流程。新用户面对平铺的 skill 列表，难以找到起点。

**建议**：创建 `docs/pipeline.md`（见下方"行动建议 P2"）。

---

### 问题 11 — Chrome 首次配置文档在多处重复（低）

`post-to-x`、`post-to-wechat`、`post-to-weibo`、`url-to-markdown`、`danger-gemini-web` 各自维护了一份配置文档，内容高度相似。`docs/chrome-profile.md` 仅说明路径，未覆盖完整首次配置流程。

**建议**：创建 `docs/chrome-setup.md`，各 skill 改为链接引用。

## 三、内容创作流程审计

### 飞轮架构（设计优良）

```
idea-radar
    ↓ ideas.jsonl
writeflow
    ↓ draft.md
format-markdown / translate / article-illustrator
    ↓ formatted.md + images
content-review
    ↓ review-report.md
multi-publish ──→ post-to-wechat / post-to-x / post-to-weibo
    ↓
post-analytics
    ↓ 写回 ideas.jsonl
idea-radar（下一轮）
```

这一闭环设计是本项目最大的架构亮点。`post-analytics` 明确提到"闭合飞轮"，表明设计意图清晰。

---

### 缺口 1 — 缺少 `post-to-xhs`（高）

`xhs-images` 生成了小红书格式图文，但没有对应的发布 skill。`post-to-wechat`、`post-to-weibo`、`post-to-x` 均有完整发布实现，小红书是唯一只有内容生成、缺少发布的平台。

---

### 缺口 2 — `idea-radar` → `writeflow` 交接未定义（中）

`idea-radar` 将评分想法追加到 `ideas.jsonl`，`writeflow` 接受"原始素材"。但没有任何文档说明如何从 `ideas.jsonl` 选出一条想法并启动写作流程（"查看清单 → 选择 idea ID → 以 source material 启动 writeflow"）。飞轮在发现与写作之间存在断层。

---

### 缺口 3 — `markdown-to-thread` → `post-to-x` 交接命令未记录（中）

`markdown-to-thread` 输出 `thread.json`，`post-to-x` 可消费线程 JSON，但两个 SKILL.md 都没有展示完整的交接命令，用户需要自行从两份技术文档中拼接流程。

---

### 缺口 4 — 无视频内容产出（中）

`yt-transcript` 实现了视频 → 文字的输入侧，但输出侧没有任何视频内容创作 skill（无抖音/Bilibili 发布，无脚本 → 视频流程）。短视频是中文创作者平台的主要内容形态。

---

### 缺口 5 — 无 Bilibili / 抖音发布（低）

Bilibili 是中文技术/知识创作者的主要长视频平台，抖音是短视频主平台，均无对应发布 skill。

---

### 缺口 6 — 无跨平台二次分发流程（低）

`multi-publish` 支持从单一源同时发布到多平台，但没有"取已发布微信文章 → 适配后发 X"的逆向流程，常见的跨平台二次分发场景无标准路径。

---

## 综合评分

| 维度 | 评分 | 主要问题 |
|------|------|----------|
| 内聚性 | 7 / 10 | 视觉 skill 缺决策指引；`danger-` 命名孤例；`post-to-wechat` 多职责 |
| 易用性 | 6 / 10 | v0.1.0 虚假 CLI；`multi-publish` 格式不一致；无入门路径 |
| 内容创作流程 | 8 / 10 | 飞轮架构优秀；缺 XHS 发布；关键交接点未文档化 |

最高价值的两项改进是 **补全 `post-to-xhs`**（结构性缺口）和 **修复 v0.1.0 Beta 标记**（用户体验即时伤害）。
